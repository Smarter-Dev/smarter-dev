For a distributed rate limiter, you must use an external state store because Lambda containers are ephemeral and have no shared memory. While you can keep a local counter in a global variable for a single “warm” container, it won’t see traffic hitting other concurrent executions, making it impossible to enforce a global limit without a centralized backbone.

The choice of store depends on how you trade off latency for persistence. Redis (ElastiCache or Momento) is the standard for high-throughput rate limiting because its in-memory nature provides sub-millisecond response times. It also supports atomic operations like INCR, which Implement a distributed rate-limiter using Momento notes is critical for ensuring counters stay accurate without complex locking.

DynamoDB is a viable alternative if you prefer a fully serverless, connectionless scaling model, but it comes with higher latency. If you go this route, you’ll need to use conditional updates or atomic increments to handle read consistency and write contention as multiple Lambda functions hit the same user’s counter simultaneously.

Before building your own, check if your entry point can handle it for you. Rate Limiting Strategies for Serverless Applications highlights that API Gateway has native throttling features that operate at the edge, which is often more cost-effective than running Lambda logic for every blocked request.

Use the CronJob resource in Kubernetes rather than running a cron daemon inside your application container. Instead of keeping a pod alive to manage its own schedule, a CronJob lets the cluster controller handle the timing and spin up a fresh pod only when it’s time to work.

For Redis session cleanup, the most resilient path is actually to stop running manual cleanup scripts entirely. Redis has native EXPIRE functionality that handles key eviction automatically. By setting a TTL (Time To Live) when you write the session key, Redis manages the memory and deletion for you, which eliminates the risk of a cleanup job failing and letting memory usage balloon.

If you must run a custom script—for example, to perform complex logic or notify other systems before deletion—configure your CronJob with a concurrencyPolicy. Setting this to Forbid prevents a new job from starting if the previous one is still running, which is a common failure mode when cleanup tasks take longer than the 10-minute interval.

Make sure your cleanup script handles signals correctly; when the job finishes or is terminated, Kubernetes sends a SIGTERM that your entrypoint needs to pass to the process to ensure a graceful exit.
