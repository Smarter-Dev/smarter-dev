To get reliable email delivery without distributed transactions, you should treat the Transactional Outbox pattern as your fundamental architectural guarantee and Change Data Capture (CDC) as the optimized mechanism for implementing it.

Think of the Outbox as a logic primitive: by writing your email task into a local database table within the same transaction as your business logic, you ensure that the task only exists if the business state is committed. This avoids the “dual-write” failure where you save a record but fail to queue the mailer, or vice-versa.

The choice between them usually boils down to your operational environment:

  • Transactional Outbox (Application-level): You write a message relay process (like a background worker) that polls your outbox table. This is easier to implement and test within the app but adds load to your database via polling and requires careful handling of race conditions (like using SELECT FOR UPDATE SKIP LOCKED in Postgres).
  • Change Data Capture (CDC): You use a tool like Debezium to stream database transaction logs directly to your message broker (e.g., Kafka). This is the “hard mode” of the pattern, but it’s significantly more efficient at scale because it removes the polling overhead and ensures that your relay is completely decoupled from your application’s query patterns.

For an agentic platform on Kubernetes, ensure your relay process—whether it’s a simple poller or a CDC connector—respects graceful shutdown signals. If your agentic pod receives a SIGTERM, it must drain its current work or checkpoint its progress so that in-flight email tasks aren’t lost or partially processed.

Because both patterns typically result in at-least-once delivery, your final hurdle is always the same: your email-sending workers must be idempotent. Even with a perfect outbox, network blips or pod restarts can cause the same email task to be dispatched twice; ensure your downstream worker checks a unique task ID before initiating any SMTP communication.

By designing your agent’s execution model to run tools in a loop, you create a predictable structure where you can anchor your observability: wrap each loop iteration in a span, attaching the tool’s name and arguments as metadata to that span.

Since agents are inherently dangerous and can trigger unintended shell commands or API calls, keeping the full context visible is non-negotiable. You’ll want to utilize distributed tracing to carry the request context through the agent’s decision process. This ensures that when a loop goes runaway, you can pivot directly from the trace view—which pinpoints exactly where the delay or error occurs—to the specific logs and metrics generated by that tool call.

Treat the agentic loop as a single distributed trace where each iteration and subsequent tool call is recorded as a nested span. By propagating the original request context into the loop, you keep the agent’s work linked to the production trace that triggered it rather than letting it drift into disconnected logs.

Since an agent’s core behavior is running tools in a loop to achieve a goal, you should map every “turn” of that loop to a trace span and every individual tool execution to a child span. This structure allows you to pinpoint exactly where slowdowns occur—revealing whether the agent is stuck in a recursive hallucination or if a specific tool is timing out. Use OpenTelemetry to standardize these spans so they carry through async boundaries and third-party API calls.

The real value of this instrumentation is the ability to unify logs, metrics, and traces in one view. When a loop goes runaway, you need to see the high-cardinality data—like the exact JSON payload passed to a tool—alongside the timing data to understand why the agent is looping. Standardizing on spans rather than just “print” debugging ensures you can observe these inherently dangerous behaviors in production without losing the thread of the original user session.
