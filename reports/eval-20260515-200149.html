Start with Event Sourcing by Martin Fowler. It’s the most durable entry point because it focuses on the data model—capturing every change as an immutable event—rather than the specific message broker you’re using.

Instead of just storing the current “snapshot” of your database, you store the sequence of things that happened. As Fowler explains, this gives you a built-in audit log; you don’t just see the current state, you see the entire history of how the system arrived there. This shift is what enables features like “time travel” debugging and high-scale read models.

When you move beyond a single service, you’ll need to understand the Pattern: Saga. Since you lose traditional ACID transactions in a distributed system, sagas coordinate multi-step workflows by chaining local transactions together. If a step fails, you trigger “compensating transactions” to undo the previous work manually.

Think of the Transactional Outbox as the architectural pattern and Change Data Capture (CDC) as one specific way to implement it. For an agentic email platform, you use the Outbox pattern to ensure that when your agent updates its internal state (like marking a draft as “ready to send”), the corresponding event (sending the email) is guaranteed to happen.

The Transactional outbox solves the “dual-write” problem by inserting your event into a dedicated outbox table within the same database transaction as your business logic. This makes the state change and the event emission atomic—one cannot succeed without the other. You then need a “Message Relay” to pick up those records and push them to your broker.

When deciding how that relay works on Kubernetes, you have a choice between Polling and CDC:

  • CDC (Log Tailing): Tools like Debezium tail the database’s transaction log (e.g., Postgres WAL) to stream changes. This is the gold standard for high-throughput systems because it’s low-latency and doesn’t put extra query load on your database.
  • Polling: A simple worker queries the outbox table for unsent rows. This is easier to get running on K8s without specialized infrastructure, but it adds load and lag.

For an agentic platform, where reliability is usually more critical than millisecond latency, starting with an application-level relay—like Transactionally Staged Job Drains—gives you the most control over retry logic and backpressure before you commit to the operational overhead of a full CDC stack.

Standard APM traces are useful for mapping service dependencies, but for agents, you specifically need high-cardinality metadata—like the exact prompt version, the tool’s return value, and the sequence index—attached to those traces. This allows you to differentiate between a healthy recursive search and a broken runaway loop by looking at the relationship between the entities and the data they pass.

You’ll want to use automated transaction tracing that captures each tool execution as a discrete span within the larger agentic loop. Since an agent is essentially running tools in a loop to reach a goal, debugging a “runaway” state requires seeing the sequence of tool inputs and outputs that triggered the next iteration.

Standard APM tools like those described in the Introduction to New Relic APM are effective here because they provide low-overhead, automatic instrumentation. By treating each tool call as a “transaction,” you can pinpoint exactly where the loop logic starts to drift or which specific “hotspot” is failing to return a terminating condition. This ensures that even if the agent enters an infinite loop, the individual traces are streamed to your collector rather than being lost when the process eventually hits a timeout or memory limit.
