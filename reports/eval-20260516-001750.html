Start with the Pattern: Transactional outbox. It’s the most important concept to learn first because it solves the “dual write” problem: the risk that you update your database but fail to notify the rest of your system (or vice versa).

The core idea is to save your “events” into a table in the same database transaction as your business data. This guarantees that if the data is saved, the event is too. A separate process then reads that table and pushes the messages to your broker. You’ll quickly realize that because this process can crash and retry, your consumers must be built to handle the same message twice—a concept known as idempotency.

Once you understand how to move data reliably, look at how services coordinate:

  • Pattern: Saga: Learn how to handle transactions that span multiple services without using slow, fragile distributed locks. It focuses on “compensating transactions”—if step three fails, you run a specific “undo” job for steps one and two.
  • Event Sourcing: This is a deeper shift where you stop storing “current state” and instead store the entire history of events. It’s powerful for auditing and “travelling back in time” to debug, but it adds significant complexity to how you query data.
  • Apache Kafka Quickstart: Use this specifically to understand the Log primitive. Unlike a traditional message queue that deletes messages after they are read, Kafka acts like a distributed, append-only file that multiple services can “replay” at their own pace.

Use SQS for 10k events/min bursts if you need to protect your primary database from IO spikes, but stick with Postgres SKIP LOCKED if you need strict atomic consistency between your data changes and your event emissions.

At 10,000 events per minute, the primary risk of using Postgres as a queue is “polling debt”—where a fleet of workers checking for new rows can effectively DDoS your database. As Celery Best Practices warns, a database isn’t a broker; high-concurrency polling can push disk IO through the roof and slow down the user-facing app sharing that same instance. However, Postgres allows you to implement the Outbox Pattern, where you insert the “work” into a table in the same transaction as your business logic, ensuring you never lose an event if the process crashes mid-request.

SQS is built for this scale and handles the “worker share” logic—distributing tasks across consumers and managing retries—without any configuration overhead. The trade-off is the “dual-write” problem: if your database update succeeds but the SQS push fails, your system becomes inconsistent. If you go the SQS route, you’ll need to design for at-least-once delivery by making your consumers idempotent, as SQS (especially standard queues) can occasionally deliver the same message twice.

Use the Transactional Outbox pattern if you need to control the exact shape of your events; use Change Data Capture (CDC) as the mechanism to move those events from your database to your message broker. For an agentic email platform, this combination ensures your agents can “read their own writes” while guaranteeing that every sent email or state change is eventually published without the fragility of distributed transactions.

The core problem you’re solving is the “dual-write” inconsistency: if you update your database and then try to hit an email API or Kafka, one will eventually fail while the other succeeds. Pattern: Transactional outbox solves this by making the “send” part of the local database transaction. You write your business data and a corresponding event record into an outbox table in one atomic step. This is especially critical for agents that might trigger multiple downstream actions based on a single prompt or state change.

While you can write a custom poller to tail that table, Reliable Microservices Data Exchange With the Outbox Pattern explains why using a CDC tool like Debezium is the more robust choice for the “relay” portion. CDC reads the database’s transaction log directly, which is more efficient than polling and ensures you don’t miss deletes or rapid updates.

The ARIES protocol is comprised of three phases. Upon start-up after a crash, the DBMS will execute the following phases: 1. Analysis: Read the WAL to identify dirty pages in the buffer pool and active transactions at the time of the crash. 2. Redo: Repeat all actions starting from an appropriate point in the log. 3. To debug an agent looping through tool calls, you should wrap each iteration in a child span and use OpenTelemetry Context Propagation to keep them tied to the original production trace. By treating each tool execution as a discrete unit of work within a larger trace, you can see the repetition patterns in real-time without losing the request context that triggered the agent.

Start by ensuring your agent’s execution loop honors the incoming trace context; in Python, this usually means ensuring your tool calls happen within the same ContextVar scope. You want to send this telemetry to a high-cardinality backend like Honeycomb or Better Stack, where you can group by specific tool names or agent IDs to spot where the logic is failing. If you’re running on Kubernetes and want to see the underlying network calls without manual code changes, Pixie can use eBPF to capture those tool-internal HTTP requests automatically.
