# Stage 02 — Persistence: DB model, migration, web API, bot service

**Stable id:** `override-persistence-api`

## Outcome
Persist a per-channel model override + daily/hourly token budgets, and expose it
over the internal web API so the bot can read/write it. Add a bot-side service the
later stages call. After this stage the data layer is fully working and tested,
but nothing in the bot UI or chat runtime uses it yet (stages 03/04).

## Prior state (from stage 01, already committed)
- `smarter_dev/bot/agents/model_catalog.py` exposes `MODEL_CATALOG`,
  `is_valid_model_key(key)`, `get_model(key)`, `catalog_by_key()`.
- DO config on `Settings`; provider router exists.

## Context you need (read before starting)
- `smarter_dev/web/models.py` — single SQLAlchemy 2.0 (async) models file. Models
  subclass `Base` (from `smarter_dev/shared/database.py`), which already supplies
  `created_at`/`updated_at`. Study existing **per-channel** bot config models for
  the exact idiom: `ChannelHandler` (`channel_handlers`, ~line 4476, keyed by
  guild_id + channel_id with a unique constraint) and `AdminHandler`
  (`admin_handlers`, ~line 4572). Discord ids are stored as `String`, not FKs.
- Alembic: **dual tree**. Bot/per-channel config lives in the **legacy** `public`
  schema. Table ownership is the `LEGACY_TABLES` frozenset in
  `alembic/legacy/env.py`. Generate legacy migrations with
  `alembic/legacy/alembic.ini`. Apply via `scripts/migrate.py`.
- Web API structure: routers in `smarter_dev/web/api/routers/`, schemas in
  `smarter_dev/web/api/schemas.py`, CRUD in `smarter_dev/web/crud.py`. Auth uses a
  `verify_api_key` dependency (bot sends `Authorization: Bearer <bot_api_key>`).
  Study the bytes-config router (per-guild config: `GET/PUT /guilds/{guild_id}/bytes/config`)
  as the closest template — find it under `smarter_dev/web/api/routers/`.
- Bot API client + services: bot calls the web API via its API client; services
  are built in `smarter_dev/bot/client.py` `setup_bot_services()` and stashed in
  `bot.d["<name>_service"]` and `bot.d["_services"]`. Study an existing service
  (e.g. the bytes/config service) for the client + optional cache pattern.
- Tests/DB: SQLite in-memory via `tests/conftest.py` (`db_session`) and
  `tests/web/test_api/conftest.py` (`real_api_client`, `bot_headers`, model tests).
- Global rules: fail fast; only catch specific exceptions; pure functions where
  possible; no inline imports; `X | None` not `Optional`; builtins for annotations.
  Always TDD.

## What to build

### 1. DB model
Add `ChannelModelOverride` to `smarter_dev/web/models.py`:
- Table `channel_model_overrides` (legacy/public schema).
- Columns:
  - `id` — surrogate PK (match the id style used by `ChannelHandler`/other legacy
    tables — likely a UUID or autoincrement int; mirror the neighbours).
  - `guild_id: Mapped[str]` (String, indexed).
  - `channel_id: Mapped[str]` (String).
  - `model_key: Mapped[str]` (String) — a `key` from `MODEL_CATALOG`.
  - `daily_token_budget: Mapped[int]` (default 0) — 0 = unlimited.
  - `hourly_token_budget: Mapped[int]` (default 0) — 0 = unlimited.
  - `created_at`/`updated_at` come from `Base`.
- **Unique constraint on `channel_id`** (one override per channel). Add an index on
  `guild_id` for listing. Follow the naming convention used by `Base`'s metadata.

Register the new table in `LEGACY_TABLES` in `alembic/legacy/env.py`.

### 2. Migration
Generate with the legacy config, then review the autogenerated file:
```
uv run alembic -c alembic/legacy/alembic.ini revision --autogenerate -m "add channel_model_overrides"
```
Verify the migration creates the table in the `public` schema with the unique
constraint + guild index, and has a correct `down_revision` chaining onto the
current legacy head. Apply and confirm:
```
uv run python scripts/migrate.py --only legacy
```
(If the working dir path contains a space, `scripts/migrate.py` is the supported
runner — do not use the `skrift db` CLI.)

### 3. Pydantic schemas
In `smarter_dev/web/api/schemas.py` add request/response models:
- `ChannelModelOverrideRead` — guild_id, channel_id, model_key, daily_token_budget,
  hourly_token_budget, created_at, updated_at.
- `ChannelModelOverrideWrite` — model_key, daily_token_budget, hourly_token_budget.
  Validators: `model_key` must satisfy `is_valid_model_key` (import from
  `model_catalog`); budgets are `int >= 0`. Reject unknown model keys / negative
  budgets with 422.

### 4. CRUD
In `smarter_dev/web/crud.py` add async functions taking an `AsyncSession`:
- `get_channel_model_override(session, guild_id, channel_id) -> ChannelModelOverride | None`
- `upsert_channel_model_override(session, guild_id, channel_id, model_key, daily, hourly) -> ChannelModelOverride`
  (insert or update the single row for that channel; do not mutate a passed-in
  object owned elsewhere — load/create inside).
- `delete_channel_model_override(session, guild_id, channel_id) -> bool`
- (optional) `list_channel_model_overrides(session, guild_id) -> list[...]`

### 5. API router
Create `smarter_dev/web/api/routers/model_overrides.py`, mounted the same way the
other routers are (find the include/registration point — likely in
`smarter_dev/web/api/app.py` or an `__init__`/router aggregator; mirror how the
bytes router is included). Endpoints, all behind `verify_api_key`:
- `GET  /guilds/{guild_id}/channels/{channel_id}/model-override` → 200
  `ChannelModelOverrideRead` or 404 if none.
- `PUT  /guilds/{guild_id}/channels/{channel_id}/model-override` (body
  `ChannelModelOverrideWrite`) → upsert, 200 with the stored row.
- `DELETE /guilds/{guild_id}/channels/{channel_id}/model-override` → 204 (idempotent).

### 6. Bot-side service
Create `smarter_dev/bot/services/model_override_service.py` with a
`ModelOverrideService` that wraps the bot API client:
- `async get_override(guild_id, channel_id) -> ChannelModelOverrideDTO | None`
- `async set_override(guild_id, channel_id, model_key, daily, hourly) -> ChannelModelOverrideDTO`
- `async clear_override(guild_id, channel_id) -> None`
Define a small DTO (dataclass in `smarter_dev/bot/services/models.py`, matching the
existing DTO idiom there). Add a short cache if the neighbouring config service
caches (mirror it; keep TTL small, e.g. 30–60s, and expose a way to invalidate on
write so stage 03's modal submit reflects immediately).

Wire it into `smarter_dev/bot/client.py` `setup_bot_services()`:
`bot.d["model_override_service"] = ModelOverrideService(...)` and add to
`bot.d["_services"]`.

## Tests (TDD — write first)
- `tests/web/test_models/` (or `tests/web/*_test.py`) — model create/read;
  unique-per-channel constraint raises on a second insert for the same channel;
  defaults (budgets default to 0).
- `tests/web/test_api/test_model_overrides.py` — using `real_api_client` +
  `bot_headers`: PUT then GET round-trips; PUT with an invalid `model_key` → 422;
  PUT with a negative budget → 422; DELETE removes it (subsequent GET → 404);
  requests without `bot_headers` → 401/403 (auth gate).
- `tests/bot/services/test_model_override_service.py` — using the
  `bot/services/conftest.py` mock API client: `set_override`/`get_override`/
  `clear_override` call the right API paths; cache invalidation on write works.
- Full suite: `uv run pytest`.

## Acceptance criteria
- `channel_model_overrides` table + migration exist; `uv run python scripts/migrate.py --only legacy`
  applies cleanly on a fresh DB.
- API round-trips and rejects invalid model keys/budgets.
- `ModelOverrideService` registered in `bot.d`.
- `uv run pytest` green. Run `semgrep` + `gitleaks` before committing.

## Notes / decisions
- Budgets are plain ints with `0 == unlimited` — enforced in stage 04, only stored
  here.
- One row per channel (unique `channel_id`); PUT is an upsert so the modal can be
  reopened to change settings.
